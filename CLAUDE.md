# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A Bun/TypeScript game-engine monorepo backed by a Rust native addon. The packages are:

- **`packages/bun-webgpu-rs`** — Rust napi-rs crate that exposes WebGPU (wgpu 24) and SDL3 to Bun/Node. See its own [`CLAUDE.md`](packages/bun-webgpu-rs/CLAUDE.md) for detailed build rules, napi-rs constraints, SDL3 quirks, and GPU pitfalls.
- **`packages/metis-game`** — TypeScript game consuming `bun-webgpu-rs` *and* `metis-engine` via `workspace:*`. Currently a 100-animated-point-light demo plus a joystick/keyboard input smoketest. It bootstraps its own window/adapter/device/surface and drives the engine without `RenderContext` — the reference consumer of metis-engine's caller-owned-device path.
- **`packages/metis-engine`** — WebGPU clustered-forward PBR renderer (HDR + auto-exposure post-processing, directional shadow mapping, HUD text via `bun-webgpu-rs`'s `VectorContext`), built on `bun-webgpu-rs`; depends on neither `metis-game` nor `metis-tui` (`metis-game` consumes *it*). Its `src/` is split into two independent subtrees: **`renderer/`** (the whole renderer, imported via `metis-engine/renderer`) and **`ecs/`** (a young archetype ECS with Structure-of-Arrays storage, imported via `metis-engine/ecs`; no `metis-data` dependency). See its own [`CLAUDE.md`](packages/metis-engine/CLAUDE.md).
- **`packages/metis-tui`** — standalone OpenTUI (`@opentui/core`/`@opentui/react`) terminal-UI ship-systems sim; doesn't depend on the other packages. See its own [`CLAUDE.md`](packages/metis-tui/CLAUDE.md).

## API docs — read these before reading source

Each of the two library packages has a **`DOC.md`**: a task-oriented API
reference (what to call, with signatures, recipes, and the invariants that break
things silently).

| Package | Doc | Ground truth |
|---|---|---|
| `bun-webgpu-rs` | [`DOC.md`](packages/bun-webgpu-rs/DOC.md) | `index.d.ts` — auto-generated by `napi build`, authoritative, never drifts |
| `metis-engine` | [`DOC.md`](packages/metis-engine/DOC.md) | the source — hand-maintained doc |
| `metis-data` | [`DOC.md`](packages/metis-data/DOC.md) | the source — hand-maintained doc |

**When working in any of these packages, read its `DOC.md` first.** It exists so you
don't have to read a dozen source files to recall an API. Fall back to source
only when the doc doesn't cover what you need, or when the task hinges on an
exact signature (for `bun-webgpu-rs`, prefer `index.d.ts` over the doc).

**Keep them current.** If you change a public API — an exported symbol's
signature, a config constant's value or meaning, a required call order, or a
documented invariant — update that package's `DOC.md` **in the same change**. A
stale doc is worse than none, because it will be trusted.

`CLAUDE.md` explains *why* (architecture, rationale, debugging history);
`DOC.md` explains *what to call*. Keep that split.

## Commands

All commands are run from the repo root unless noted.

```powershell
# Install all dependencies
bun install

# Build the Rust native addon (debug, from packages/bun-webgpu-rs)
cd packages/bun-webgpu-rs && bun run build:debug

# Build release
cd packages/bun-webgpu-rs && bun run build

# Run all tests (from bun-webgpu-rs)
cd packages/bun-webgpu-rs && bun test

# Run a single test file
cd packages/bun-webgpu-rs && bun test test/render-window.test.ts --timeout 15000

# Type-check (root, covers metis-game via paths alias)
bunx tsc --noEmit

# Type-check bun-webgpu-rs binding types only
cd packages/bun-webgpu-rs && bunx tsc --noEmit

# Run metis-game
bun run packages/metis-game/src/index.ts
```

## Architecture

```
metis-wgpu/
  packages/
    bun-webgpu-rs/       Rust cdylib → .node file loaded by index.js
      src/
        lib.rs           Re-exports all public symbols; edit when adding new ones
        gpu/             WebGPU types: instance, adapter, device, surface, encoder, pipeline, …
        sdl/             SDL3 bindings: init, window, events, keyboard, mouse, joystick, gamepad
      index.js           Auto-generated by napi-rs; do not edit
      test/              Bun test files (.test.ts)
    metis-game/
      src/index.ts        Game entry point; a 100-point-light demo + joystick smoketest.
                          Owns its own window/adapter/device/surface and drives
                          metis-engine directly (metis-engine/DOC.md §1.3) — the
                          reference consumer of the caller-owned-device path.
```

### Data flow

1. Rust compiles to `bun-webgpu-rs.win32-x64-msvc.node` (or platform equivalent).
2. `index.js` (auto-generated by napi-rs) loads the `.node` file and re-exports all napi symbols.
3. `metis-game` imports from `bun-webgpu-rs` which resolves to `index.js`.

### Key conventions

- The root `tsconfig.json` maps `"bun-webgpu-rs"` to `./packages/bun-webgpu-rs` so the type-checker sees the `.d.ts` files without building.
- `napi-rs` is the only FFI layer — no raw N-API calls in JS.
- After any change to Rust source, rebuild with `bun run build:debug` before running tests or the game.
