# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A Bun/TypeScript game-engine monorepo backed by a Rust native addon. The packages are:

- **`packages/metis-native`** â€” Rust napi-rs crate that exposes WebGPU (wgpu 24) and SDL3 to Bun/Node. Close to the WebGPU spec, with a small, explicitly-typed set of native-only wgpu extensions (`GPUNativeFeatureName`). See its own [`CLAUDE.md`](packages/metis-native/CLAUDE.md) for detailed build rules, napi-rs constraints, SDL3 quirks, and GPU pitfalls.
- **`packages/metis-game`** â€” TypeScript game consuming `metis-native` *and* `metis-engine` via `workspace:*`. Currently a 100-animated-point-light demo plus a joystick/keyboard input smoketest. It bootstraps its own window/adapter/device/surface and drives the engine without `RenderContext` â€” the reference consumer of metis-engine's caller-owned-device path.
- **`packages/metis-engine`** â€” WebGPU clustered-forward PBR renderer (HDR + auto-exposure post-processing, directional shadow mapping, HUD text + debug graph/tree widgets via `metis-native`'s `VectorContext`, and an opt-in timestamp-query GPU profiler), built on `metis-native`; depends on neither `metis-game` nor `metis-tui` (`metis-game` consumes *it*). Its `src/` is split into two independent subtrees: **`renderer/`** (the whole renderer, imported via `metis-engine/renderer`) and **`ecs/`** (a young archetype ECS with Structure-of-Arrays storage, imported via `metis-engine/ecs`; no `metis-data` dependency). See its own [`CLAUDE.md`](packages/metis-engine/CLAUDE.md).
- **`packages/metis-tui`** â€” standalone OpenTUI (`@opentui/core`/`@opentui/react`) terminal-UI ship-systems sim; doesn't depend on the other packages. See its own [`CLAUDE.md`](packages/metis-tui/CLAUDE.md).

## API docs â€” read these before reading source

Each of the two library packages has a **`DOC.md`**: a task-oriented API
reference (what to call, with signatures, recipes, and the invariants that break
things silently).

| Package | Doc | Ground truth |
|---|---|---|
| `metis-native` | [`DOC.md`](packages/metis-native/DOC.md) | `index.d.ts` â€” auto-generated by `napi build`, authoritative, never drifts |
| `metis-engine` | [`DOC.md`](packages/metis-engine/DOC.md) | the source â€” hand-maintained doc |
| `metis-data` | [`DOC.md`](packages/metis-data/DOC.md) | the source â€” hand-maintained doc |

**When working in any of these packages, read its `DOC.md` first.** It exists so you
don't have to read a dozen source files to recall an API. Fall back to source
only when the doc doesn't cover what you need, or when the task hinges on an
exact signature (for `metis-native`, prefer `index.d.ts` over the doc).

**Keep them current.** If you change a public API â€” an exported symbol's
signature, a config constant's value or meaning, a required call order, or a
documented invariant â€” update that package's `DOC.md` **in the same change**. A
stale doc is worse than none, because it will be trusted.

`CLAUDE.md` explains *why* (architecture, rationale, debugging history);
`DOC.md` explains *what to call*. Keep that split.

## Commands

All commands are run from the repo root unless noted.

```powershell
# Install all dependencies
bun install

# Build the Rust native addon (debug, from packages/metis-native)
cd packages/metis-native && bun run build:debug

# Optimized build for iterating on performance (default `build`, `fastdev`
# profile: opt-level 3, no LTO). ~19 s to rebuild after a one-line change,
# vs ~103 s for `build:release` â€” use this when profiling.
cd packages/metis-native && bun run build

# Real release build (fat LTO). Slow; for shipping, not for iterating.
cd packages/metis-native && bun run build:release

# Run all tests (from metis-native)
cd packages/metis-native && bun test

# Run a single test file
cd packages/metis-native && bun test test/render-window.test.ts --timeout 15000

# Type-check (root, covers metis-game via paths alias)
bunx tsc --noEmit

# Type-check metis-native binding types only
cd packages/metis-native && bunx tsc --noEmit

# Run metis-game
bun run packages/metis-game/src/index.ts
```

## Architecture

```

  packages/
    metis-native/       Rust cdylib â†’ .node file loaded by index.js

```

### Data flow

1. Rust compiles to `metis-native.win32-x64-msvc.node` (or platform equivalent).
2. `index.js` (auto-generated by napi-rs) loads the `.node` file and re-exports all napi symbols.
3. `metis-game` imports from `metis-native` which resolves to `index.js`.

### Key conventions

- The root `tsconfig.json` maps `"metis-native"` to `./packages/metis-native` so the type-checker sees the `.d.ts` files without building.
- `napi-rs` is the only FFI layer â€” no raw N-API calls in JS.
- After any change to Rust source, rebuild with `bun run build:debug` before running tests or the game.
